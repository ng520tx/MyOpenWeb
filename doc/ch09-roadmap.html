SECTION 9.1先定位：市场要什么样的人

2026 年招聘市场的三个信号（来自公开岗位数据分析）对你都是利好：

  • 78% 的 AI 应用岗面向 5 年以上经验的工程师——行业要的不是应届算法生，而是「工程功底扎实 + 会用 LLM 构建系统」的人。你 19 年的工程经验是入场券而不是包袱。
  • RAG 出现在约 18% 的企业岗位 JD 里，prompt 工程需求年增 227%——你项目的主线能力正是市场主流需求。
  • 不需要模型训练背景：AI 应用工程师（AI Engineer）用预训练模型 API 构建产品，与研究岗（炼模型）是两条职业线。别被「不会炼丹」吓退，岗位画像里根本没有这一项。
你的定位公式

「资深工程师 × AI 应用能力」的交集人群在 2026 年仍然稀缺。面试叙事永远从这个交集讲：不是「我转行学了个新东西」，而是「我把 19 年的工程方法论应用到了 AI 应用这个新领域」——分层、测试、评测、降级、可观测，全是你的旧武器打的新仗。

SECTION 9.3RAG 前沿：2026 年生产界的共识与新边界

2026 年中的行业共识（多份生产实践指南交叉验证）：

  • 「单向量索引 + top-k」的裸 RAG 已死：真实企业语料上检索失败率高达 40%，不再是任何认真场景的参考架构。生产默认栈 = 混合检索（BM25+向量）→ rerank（宽召回精排）→ 带引用生成——恭喜，这正是你项目已经实现的全套。
  • Agentic RAG 是「升级」不是「默认」：业界口径与你的实现完全一致——先跑线性管道、评测找到失败的查询类别、只对那些类别引入模型参与的检索决策。你的 Grader 有界补检正是这个哲学。
  • CRAG（Corrective RAG）成为标准词汇：先给检索结果打分、不合格就改写重检——你项目的「检索自纠错」就是 CRAG 思想的实现，面试请主动用这个术语。

接下来的三个新边界，是你「下一个项目迭代」的候选方向：

方向解决什么2026 状态落到你项目
GraphRAG跨文档综合题（「总结这 200 份合同的共同风险」）——单片检索天然答不了微软开源实现成熟；生产模式是路由分流：小分类器把「点查题」发向量 RAG、「综合题」发图 RAG从运维文档抽取实体关系（服务→依赖→告警）建轻量图，先做「服务依赖问答」一个场景
多模态 RAG图表/截图/表格里的信息检索（运维场景大量架构图、监控截图）早期生产，多模态 embedding（联合编码文本+图像）快速成熟中你已有 OCR 链路，下一步是图片直接进向量索引
pgvector 迁移十万级以上语料的检索性能 + SQL 生态统一HNSW 索引成熟，PostgreSQL 生态事实标准repositories 层已隔离 SQL，迁移是受控改动（第 8 章 Q5 的演进路径）

评测生态同样值得跟进：RAGAS（你自研 judge 对齐的就是它的指标体系）、TruLens、Opik——你的路线是「原理已自研验证 → 生产直接上框架」，和 Agent 框架同一个故事。

面试官视角 · 本章问答

Q1你没用 LangChain，那你了解这个生态吗？▸

「了解且实际跑过。2026 年的格局：LangGraph 是生产标准（v1.0 后成为 LangChain 官方运行时，状态图 + checkpoint + human-in-the-loop），CrewAI 是角色制快速原型，AutoGen 分裂成社区 AG2 和微软 Agent Framework（Azure 系首选）。我仓库里有 LangGraph 双工具 agent 的可运行示例，概念映射我很清楚：我的事件生成器循环 = 它的图节点执行，我的 agent_steps 落库 = 它的 checkpointer，我的 SSE 事件 = 它的 stream 模式。自研让我知道框架在替我做什么，所以上手任何一家都是语法成本而不是概念成本。」

Q2如果新项目让你选型，你选框架还是自研？▸

「默认选框架，LangGraph 优先——团队协作需要社区文档和人才池，checkpoint/observability 这些轮子没必要重造。两种例外我会自研：①输出契约极窄的单一场景（一个固定三步的流水线，上框架是杀鸡用牛刀）；②极端环境约束（离线内网、依赖审计严格——我的 MyOpenWeb 就是这类）。关键是我两条路都走得了，选型依据是约束而不是熟悉度。」

Q3GraphRAG 是什么？什么时候需要它？▸

「解决单片检索答不了的跨文档综合题。普通 RAG 把语料当独立片段的平面集合，问『这 200 份合同的共同风险』时，答案不在任何一个片段里。GraphRAG 先从文档抽实体和关系建知识图谱，检索时沿关系遍历、对相关实体社区做摘要。2026 年的生产模式是路由：小分类器把点查题发给便宜快速的向量 RAG，综合题才走昂贵的图 RAG——不是替代是分流。我的运维场景里『服务 A 依赖谁、挂了影响谁』就是典型图问题，这是我项目的下一个迭代方向。」

Q4AI 应用变化这么快，你怎么保持更新？▸

「一手来源优先 + 落到代码验证。固定跟：Anthropic/OpenAI 的工程博客（Building Effective Agents 这类）、MCP 规范更新、LangChain changelog、RAGAS 文档。判断信息价值的过滤器是『能不能回灌到我的项目』——比如我看到 CRAG 论文后把检索自纠错做进了系统并建了困难集评测，这比收藏一百篇文章有用。我给自己的节奏是每月一个可 commit 的新实践。」

本章小结 · 装进口袋的 5 句话
  • 市场要「资深工程 × AI 能力」的交集，你的叙事是旧武器打新仗，不是转行从零。
  • 框架地图：LangGraph 生产标准 / CrewAI 快速原型 / MS Agent Framework 挺进 Azure 系——懂原理的人换框架只是换语法。
  • 2026 RAG 共识：混合+rerank 是默认栈（你已达标）；Agentic/CRAG 是按失败类别引入的升级（你已实现）；GraphRAG 用路由分流接综合题（你的下一步）。

  • 精进防坑：每一分学习必须变成 commit 或简历行，拒绝囤课。
