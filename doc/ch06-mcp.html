SECTION 6.1解决什么问题：M×N 的集成灾难

2023–2024 年的乱象：每个 AI 客户端（Cursor、Claude Desktop、各家 Copilot）想接每个工具源（GitHub、数据库、内部知识库），都要写一套私有插件。M 个客户端 × N 个工具 = M×N 套适配代码，且互不通用。

Anthropic 在 2024 年 11 月开源 MCP，把中间那条线标准化：工具方实现一次 MCP Server，客户端实现一次 MCP Client，任意组合即插即用——集成成本从 M×N 降到 M+N。2025 年 OpenAI、Google 相继采纳，它成了事实标准。

前端类比

USB-C 之前，每个手机厂一个充电口，线缆厂商要为每种口各出一根线；USB-C 把「口」标准化后，任何线配任何设备。再贴近你的经验一点：MCP 之于 AI 工具，就像 LSP（Language Server Protocol）之于编辑器——LSP 出现前每个编辑器为每种语言写一遍补全，之后语言方实现一个 Language Server 就通吃所有编辑器。MCP 抄的正是 LSP 的作业，连 JSON-RPC 底座都一样。

SECTION 6.2协议本质：三层拆解

层内容你已有的对应知识
消息层JSON-RPC 2.0：请求带 id/method/params，响应带 id/result形状像 Redux action / 老式 XML-RPC 的 JSON 版
传输层stdio（客户端把 Server 当子进程拉起，读写标准输入输出）或 Streamable HTTP（远程部署）stdio ≈ 管道 ps | grep；HTTP 版 ≈ 普通 Web 服务
能力层三种原语：tools（模型可调用的函数）、resources（可读数据，类似 GET）、prompts（预置提示词模板）tools ≈ POST 接口；resources ≈ 静态资源路由

一次完整会话的报文流（以 Cursor 连本项目为例）：

→ {"id": 0, "method": "initialize", "params": {"protocolVersion": "..."}}   // 握手
← {"id": 0, "result": {"capabilities": {"tools": {}}, "serverInfo": {"name": "myopenweb"}}}

→ {"id": 1, "method": "tools/list"}                                        // 问有什么工具
← {"id": 1, "result": {"tools": [{"name": "search_knowledge",
     "description": "在指定企业知识库中检索资料片段…",
     "inputSchema": {"type": "object", "properties": {"knowledge": {...}, "query": {...}}}}, ...]}}

→ {"id": 2, "method": "tools/call",                                        // 调用
   "params": {"name": "search_knowledge",
              "arguments": {"knowledge": "运维手册", "query": "Redis 默认端口"}}}
← {"id": 2, "result": {"content": [{"type": "text", "text": "{\"results\": [...]}"}]}}
  • 就三个动词：initialize 握手、tools/list 拿说明书、tools/call 执行。第 5 章 function calling 的 tools 数组，内容正是从 tools/list 的返回喂给模型的。
  • stdio 传输下没有端口、没有 URL——客户端 spawn 子进程，JSON-RPC 报文一行一条走 stdin/stdout。这也是为什么调试 MCP Server 不能 print 到 stdout（会污染协议流），日志必须走 stderr。

SECTION 6.3读代码：130 行的完整 MCP Server

SDK（FastMCP）把协议层全部吞掉，你写的只剩业务。看本项目把知识库检索暴露成 MCP 工具的全部代码：

from mcp.server.fastmcp import FastMCP

mcp = FastMCP("myopenweb")                # Server 名字，客户端握手时可见

@mcp.tool()
async def search_knowledge(knowledge: str, query: str, top_k: int = 4) -> dict[str, Any]:
    """在指定企业知识库中检索资料片段（混合检索：BM25 + 向量 RRF 融合）。

    Args:
        knowledge: 知识库名称或 id（可先用 list_knowledge_bases 查看）
        query: 检索问题或关键词
        top_k: 返回片段数量，默认 4
    """
    knowledge_id = _resolve_knowledge_id(knowledge)
    if not knowledge_id:
        names = [item.name for item in list_knowledge()]
        return {"error": f"知识库 '{knowledge}' 不存在", "available": names}

    config = get_provider_config()
    chunks = await query_knowledge(          # ← 完整复用第4章的检索链路
        config, config.embedding_model, knowledge_id, query,
        top_k=max(1, min(top_k, 10)),
    )
    ...

@mcp.tool()
def analyze_log(log: str) -> dict[str, Any]:
    """分析应用/运维日志：统计级别分布、抽取错误行与异常类型、定位首尾错误。"""
    return run_tool("analyze_log", {"log": log})    # ← 一行包装第5章的工具

def main() -> None:
    init_db()
    mcp.run()                              # 默认 stdio transport，开始收发 JSON-RPC
  • schema 是自动生成的：函数的类型标注（knowledge: str、top_k: int = 4）变成 inputSchema 的字段类型和默认值；docstring 变成工具 description、Args 段变成参数 description。代码即文档即协议——第 1 章学的类型标注在这里兑现了第三种价值。
  • _resolve_knowledge_id 的容错值得注意：调用方传名字或 id 都行，不存在时返回可用列表而不是干巴巴的报错——错误信息设计成「模型看了知道下一步怎么办」，与第 5 章工具的 hint 思想一脉相承。
  • top_k=max(1, min(top_k, 10))：外部输入夹紧到安全范围，边界防御只出现在系统边界（协议入口），内部函数互相信任——这是全项目一致的防御哲学。

SECTION 6.4架构故事：业务与协议解耦

注意 analyze_log 那个工具只有一行——因为业务逻辑在 agent_tools.py 里是纯函数，MCP 只是它的第二个「插座」。整个项目此刻有三个协议入口共用同一套业务核心：

入口协议消费者适配层代码量
FastAPI routersHTTP REST + SSE自家 H5 / Android 壳routers/ 薄壳
agent_runner内部函数调用自家 Agent 循环run_tool 分发
mcp_serverMCP (JSON-RPC/stdio)Cursor / Claude Desktop / 任意 MCP 客户端130 行装饰器包装
面试话术（原样可用）

「我加 MCP Server 只花了 130 行，因为检索和工具在设计时就是与协议无关的纯函数/服务——HTTP、Agent 内部调用、MCP 是三个平行的薄适配层。以后出现新协议，再加一层就是了。这验证了我分层设计的正确性。」——用一个具体数字（130 行）证明一个抽象原则（解耦），是最有说服力的讲法。

SECTION 6.5MCP 与 function calling：不是二选一

高频混淆点，一张图分清：

  模型(LLM)  ←── function calling ──→  你的应用(Agent 编排)  ←── MCP ──→  工具服务
   "我想调 search_knowledge"              把工具清单给模型；            标准化的工具发现
    (决策，模型↔应用之间)                  把调用转发给工具方             与执行(应用↔工具之间)
  • function calling 解决「模型如何表达调用意图」——模型和应用之间的约定（第 5 章）。
  • MCP 解决「应用如何发现和调用外部工具」——应用和工具服务之间的约定（本章）。
  • 典型闭环：应用启动 → MCP tools/list 拿到工具清单 → 转成 tools 数组喂给模型 → 模型 function calling 说要调 → 应用翻译成 MCP tools/call 执行 → 结果回填。两个协议在同一条链路的不同段上接力。

面试官视角 · 本章问答

Q1什么是 MCP？解决什么问题？▸

「Anthropic 2024 年底开源的开放协议，标准化 AI 应用与外部工具/数据源之间的连接。解决 M×N 集成灾难：以前每个 AI 客户端接每个工具源都要写私有适配，MCP 把接口标准化后降到 M+N——工具方实现一次 Server，所有支持 MCP 的客户端即插即用。技术上是 JSON-RPC 2.0 + stdio/HTTP 传输 + tools/resources/prompts 三种能力原语，思路直接借鉴了 LSP。我的项目实现了一个真实的 MCP Server，把企业知识库检索暴露出去，Cursor 可以直连调用。」

Q2MCP 和 function calling 什么关系？▸

「不同层，互补不竞争。function calling 是模型↔应用之间的约定：模型怎么表达『我要调工具』；MCP 是应用↔工具服务之间的约定：应用怎么发现、调用外部工具。完整链路是接力：MCP tools/list 拿清单 → 转成 tools 参数喂模型 → 模型 function calling 表达意图 → 应用翻译成 MCP tools/call 执行。我的项目里两层都有真实实现，Agent 循环走 function calling，对外服务走 MCP。」

Q3你的 MCP Server 怎么实现的？花了多少工作量？▸

「130 行。用官方 Python SDK 的 FastMCP：@mcp.tool() 装饰业务函数，SDK 从类型标注生成 inputSchema、从 docstring 生成 description，协议握手和 JSON-RPC 编解码全部托管。工作量小的根本原因是架构：检索链路和工具函数本来就是协议无关的纯函数，MCP 只是第三个薄适配层（前两个是 HTTP routers 和 Agent 内部调用）。这130行里我还是做了三件正经事：名称/ID 双向解析、错误信息带可用列表（让模型知道怎么纠正）、top_k 边界夹紧。」

Q4stdio 和 HTTP 两种 transport 怎么选？▸

「stdio：客户端把 Server 当子进程拉起，报文走标准输入输出——零网络配置、天然单用户隔离、进程随客户端生灭，本地个人工具首选，我目前就用它（Cursor 通过 wsl.exe 拉起）。Streamable HTTP：Server 作为常驻网络服务，支持远程访问、多客户端共享、可以加认证和网关——团队共享部署用它。我的迁移路径清晰：FastMCP 两种 transport 都支持，业务代码零改动，改一个启动参数加一层鉴权就能变成团队服务。」

追问预判：「stdio 模式下怎么打日志？」→ 答：stdout 是协议通道，print 会污染 JSON-RPC 流导致解析失败；日志必须走 stderr 或文件。这是 MCP 开发第一坑，我踩过。

Q5MCP 生产化有什么安全问题要考虑？▸

「四层：认证授权——HTTP transport 要加 OAuth/API-Key，工具级还要权限分级（检索只读 vs 写操作）；输入校验——SDK 只保证类型正确，业务约束（top_k 上限、路径白名单）要自己夹紧；注入风险——工具返回的内容会进入模型上下文，恶意文档里可能藏『忽略之前的指令』这类 prompt 注入，需要内容清洗或标记来源隔离；供应链——第三方 MCP Server 相当于给模型装了任意插件，企业环境要白名单管控可接入的 Server。我的当前实现定位内网单用户，前两层已做，后两层在 roadmap。」

本章小结 · 装进口袋的 5 句话
  • MCP 把 AI 工具集成从 M×N 降到 M+N，是 LSP 思路在 AI 时代的复刻。
  • 协议栈 = JSON-RPC 2.0 + stdio/HTTP transport + tools/resources/prompts 原语；核心动词就三个：initialize、tools/list、tools/call。
  • FastMCP 装饰器：类型标注 → inputSchema，docstring → description，代码即协议。
  • function calling 管模型↔应用，MCP 管应用↔工具，同链路不同段的接力。
  • 130 行接入新协议 = 业务纯函数 + 薄适配层的架构红利，这是可以现场演示的面试杀手锏。
