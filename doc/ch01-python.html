SECTION 1.1心态设定：Python 对你来说是「第五门方言」

你已经会 Java、TypeScript、JavaScript、Kotlin。Python 和它们同属「命令式 + 面向对象 + 一点函数式」家族，90% 的概念完全一致，只是拼写不同。真正的差异只有三处：

  • 缩进即语法：没有 { }，用 4 空格缩进划分代码块。你写惯了 Prettier 格式化的代码，这其实是把你已有的习惯变成了强制规定。
  • 动态类型 + 可选标注：默认像 JS 一样松，但现代 Python（含本项目）全部写类型标注，读起来和 TS 几乎一样。
  • 没有编译步骤：python xx.py 直接跑，改完即生效（≈ ts-node / nodemon 的体验）。
前端类比

把 Python 当成「没有花括号的 TypeScript + 自带标准库的 Node」。pip ≈ npm，venv ≈ node_modules 的隔离思想，requirements.txt ≈ package.json 的 dependencies 字段，uvicorn ≈ node server.js 那个进程。

SECTION 1.4四个「本项目读码关键」语法点

下面四个语法点在 server/ 里密集出现，每个都配真实代码。啃完这四个，源码就没有读不懂的行了。

① 装饰器 @xxx —— 就是 Java 注解 / NestJS 装饰器

装饰器本质是「一个接收函数、返回加工后函数的高阶函数」，写在 @ 后面挂到目标上方。你在 NestJS 里写过 @Get()、在 Java 里写过 @GetMapping，是同一个思想：用声明式标记给函数附加行为。

@router.get("", response_model=MemoriesResponse)
def get_memories() -> MemoriesResponse:
    return MemoriesResponse(memories=list_memories())
  • @router.get("")：把下面这个函数注册为 GET /api/memories 的处理器——和 NestJS 的 @Get() 一字不差的思想。
  • -> MemoriesResponse：返回类型标注，同时告诉 FastAPI 用这个模型序列化响应（详见第 2 章）。

② with 语句 —— 自动释放资源的 try-finally

Java 有 try-with-resources，Kotlin 有 use { }，Python 的对应物是 with。进入代码块时拿资源，无论正常退出还是抛异常都保证释放。

def list_memories() -> list[Memory]:
    with get_db() as conn:
        rows = conn.execute(
            """
            SELECT * FROM memories
            ORDER BY enabled DESC, updated_at DESC
            """
        ).fetchall()
    return [_row_to_memory(row) for row in rows]
  • with get_db() as conn:：拿到数据库连接，代码块结束时自动 commit + close（实现见下一个语法点）。
  • """..."""：三引号多行字符串 ≈ TS 的反引号模板串，这里用来写 SQL。
  • 最后一行就是刚才说的列表推导式：把每行数据库记录转成 Memory 模型，等价 rows.map(_rowToMemory)。

③ yield 与 @contextmanager —— 生成器的两种用途

yield 和 JS 的 generator function 里的 yield 是同一个东西：函数执行到这里「暂停、交出一个值，等下次再继续」。本项目用它实现了上面 get_db() 的「自动释放」：

@contextmanager
def get_db() -> Iterator[sqlite3.Connection]:
    conn = _connect()
    try:
        yield conn        # ← with 块里的代码在这一行"中间"执行
        conn.commit()
    finally:
        conn.close()
  • 读法：yield 之前 = 进入 with 时执行（建连接）；yield 之后 = 退出 with 时执行（提交）；finally 保证连接必然关闭。
  • 这个模式在前端最接近的东西是 React 的 useEffect(() => { setup; return cleanup })——一个函数同时描述「进场」和「离场」。
  • 第 2 章讲 SSE 时你会看到 yield 的另一个用途：一边生成数据一边发给客户端（流式响应），和这里的资源管理是同一个语法、不同的用法。

④ async / await —— 和 JS 几乎一样，但没有 Promise 对象

Python 3.5+ 的 async def / await 和 JS 语义相同：单线程事件循环上的协作式并发。差异只在生态用词：JS 的 Promise ≈ Python 的 coroutine / Task，Promise.all ≈ asyncio.gather。

@router.post("/chat/completions")
async def chat_completions(payload: ChatCompletionRequest):
    config = get_provider_config()
    data = payload.model_dump()

    sources: list[dict[str, Any]] = []
    if data.get("knowledge_id"):
        system_prompt, sources = await retrieve_for_chat(config, data)
        data["system_prompt"] = system_prompt

    response = await create_chat_completion(config, data)
    return response
  • async def 的路由函数可以 await 网络调用（请求 Ollama），期间事件循环去处理别的请求——和 Node 的非阻塞 IO 完全同一模型。
  • system_prompt, sources = await ...：函数返回元组，左边直接解构，≈ const [prompt, sources] = await ...。
  • data.get("knowledge_id")：字典安全取值，键不存在返回 None 而不是抛错，≈ data["knowledge_id"] ?? null。
与 JS 的关键差异

JS 里调用 async 函数「不 await 也会执行」（返回 pending Promise）；Python 里 coroutine 不 await 就完全不会执行，只会得到一个协程对象和一条警告。忘写 await 是 Python 新手第一大坑。

SECTION 1.5实战读码：把 memories 仓储层从头读到尾

现在你已具备全部语法知识。打开 server/repositories/memories.py（102 行），对照下面的导读自己读一遍：

def _now_ms() -> int:
    return int(time.time() * 1000)          # ≈ Date.now()

def _row_to_memory(row) -> Memory:          # ≈ 私有的 rowMapper
    return Memory(
        id=row["id"],
        content=row["content"],
        category=row["category"],
        enabled=bool(row["enabled"]),        # SQLite 存 0/1，转 bool
        created_at=row["created_at"],
        updated_at=row["updated_at"],
    )

def create_memory(payload: MemoryCreate) -> Memory:
    now = _now_ms()
    memory_id = uuid.uuid4().hex             # ≈ crypto.randomUUID()
    with get_db() as conn:
        conn.execute(
            "INSERT INTO memories (id, content, category, enabled, created_at, updated_at) "
            "VALUES (?, ?, ?, ?, ?, ?)",
            (memory_id, payload.content.strip(), payload.category,
             1 if payload.enabled else 0, now, now),
    )
        row = conn.execute("SELECT * FROM memories WHERE id = ?", (memory_id,)).fetchone()
    return _row_to_memory(row)
  • 整个文件没有类、没有框架魔法——就是「函数 + SQL + 模型转换」。这一层等价于你在 Java 里写过无数遍的 DAO。
  • VALUES (?, ?, ?) 参数化查询防 SQL 注入，与 JDBC PreparedStatement 同理。
  • Memory(id=..., content=...)：Pydantic 模型用命名参数构造，≈ new Memory({ id, content })，且构造时自动做类型校验（第 2 章展开）。

面试官视角 · 本章问答

Q1你主力是前端/Android，Python 能到什么程度？▸

参考答案框架：不要说「会一点」。说：「我的 Python 是在做 MyOpenWeb 后端时练出来的——FastAPI 分层、SQLite 仓储、asyncio 流式、pytest 都实际写过。语言本身对我不是门槛，我 19 年写过 Java/TS/Kotlin，Python 是第五门方言；真正花时间的是 AI 应用的领域知识，这正是这个项目的价值。」

追问预判：「asyncio 和 Node 事件循环有什么区别？」→ 答：模型同为单线程协作式调度；差异是 Python 里 coroutine 必须 await 才执行、且同步阻塞调用（如 requests 库）会卡死整个循环，所以本项目网络调用都走异步或线程池。

Q2Python 的 GIL 是什么？对你的服务有影响吗？▸

GIL（全局解释器锁）：CPython 同一时刻只允许一个线程执行 Python 字节码，所以多线程无法利用多核跑 CPU 密集任务。

结合项目答：「我的服务是典型 IO 密集型——大部分时间在等 Ollama 推理返回，GIL 期间会释放，asyncio 单进程就能扛并发。真正 CPU 重的部分（embedding 向量计算）交给 numpy，它在 C 层执行也会释放 GIL。如果要横向扩容，uvicorn 可以开多 worker 进程绕开 GIL。」

Q3解释一下你项目里 get_db() 的实现原理。▸

「@contextmanager 装饰的生成器函数：yield 前建立 SQLite 连接（进入 with 时执行），yield 把连接交给业务代码，业务代码跑完回到 yield 之后 commit，finally 里保证 close。等价 Java 的 try-with-resources。好处是仓储层每个函数都不可能忘记释放连接。」

追问预判：「每个请求都开新连接，性能可以吗？」→ 答：SQLite 是进程内嵌入式库，"连接"只是打开文件句柄，开销微小；换 PostgreSQL 时才需要引入连接池（如 asyncpg pool），这是我 roadmap 里 pgvector 迁移的一部分。

Q4Python 类型标注是强制的吗？你们怎么保证类型安全？▸

「运行时不强制——标注本身只是元数据，解释器不检查。但有两层保障：静态层用 mypy/Pyright 这类 checker 在 CI 或 IDE 里查（≈ tsc --noEmit）；运行时层在系统边界用 Pydantic 真校验——请求体不符合模型直接 422。我的项目所有 API 入参出参都过 Pydantic，内部函数靠标注 + IDE 提示。」

Q5列表推导式和 map/filter 你选哪个？为什么？▸

「Python 社区惯例是推导式优先：[f(x) for x in xs if cond(x)] 一眼看清取什么、从哪来、什么条件；Python 的 map/filter 返回惰性迭代器还要包 list()，反而啰嗦。逻辑复杂到推导式一行放不下时，就老老实实写 for 循环——可读性大于炫技。」

本章小结 · 装进口袋的 6 句话
  • Python = 没有花括号的 TS：def f(x: int) -> str 读起来和 TS 函数签名一回事。
  • venv ≈ node_modules，pip ≈ npm，requirements.txt ≈ package.json，uvicorn ≈ dev server。
  • 列表推导式 = filter().map() 压成一行，读法「取什么 / 从哪来 / 什么条件」。
  • 装饰器 @ = Java 注解 / NestJS 装饰器：声明式给函数挂行为。
  • with + @contextmanager（yield 分隔进场/离场）= try-with-resources。
  • Python 的 async/await 同 JS，唯一大坑：coroutine 不 await 就根本不执行。
