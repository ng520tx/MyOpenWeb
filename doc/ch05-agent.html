CH 05第二部分 · 核心原理

Agent 工具调用循环
—— prompt 与 native 双协议

「Agent」这个词被营销吹得神乎其神，实现上它是：一个最多转 3 圈的 while 循环，循环体里问模型「下一步干什么」，模型要么调工具、要么给最终答案。这一章逐段读 agent_runner.py（561 行，全项目最核心的文件），看懂之后你会发现所有 Agent 框架的内核都是这个循环。

SECTION 5.1去魅：Agent 是一个带出口的循环

LLM 只会生成文本，它不能查数据库、算数学、发请求。Agent 的思路：让模型生成「我想调用什么工具」的声明，由你的代码真正执行，把结果喂回去，模型再决定下一步。画成伪代码：

messages = [用户问题]
for round in range(3):                    # 硬上限，防失控
    decision = llm(system_prompt + messages)   # 问模型：下一步？
    if decision.action == "final":
        return decision.answer            # 出口①：模型说答完了
    for call in decision.tool_calls:
        result = run_tool(call.name, call.parameters)   # 我们执行，不是模型
        messages.append(工具结果)          # 回填，进入下一圈
return "超过最大步数"                       # 出口②：兜底
前端类比

这就是一个 redux 循环：模型是 reducer（纯函数，输入状态输出 action），你的代码是中间件（真正执行副作用），messages 数组是 store（唯一状态源）。模型永远不直接产生副作用——它 dispatch「我要调 calculator」这个 action，副作用在你可控的代码里发生。这个「声明与执行分离」是 Agent 安全性的根基。

面试一句话

「Agent = LLM 在循环里反复做结构化决策，工具由后端白名单执行，结果回填对话，直到模型给出最终答案或触达步数上限。」——能把「上限」「白名单」「回填」三个词说进去，就和背概念的人拉开差距了。

SECTION 5.2协议一：prompt 协议 —— 任何模型都能跑

模型输出的是自由文本，怎么让它「结构化决策」？第一种办法简单粗暴：在 system prompt 里把协议写成说明书。看本项目的真实指令：

AGENT_INSTRUCTION = """你是 MyOpenWeb 的 Agent v1。
你可以使用后端提供的安全工具，但不能编造工具结果。

可用工具：
{tools}                                  ← 运行时注入工具清单(JSON Schema)

你每次只能输出一个 JSON 对象，不能输出 Markdown，不能输出代码块。

如果需要调用工具，输出 Open WebUI 风格的 tool_calls 数组：
{{"tool_calls":[{{"name":"工具名","parameters":{{...}}}}]}}

如果可以最终回答，输出：
{{"action":"final","answer":"最终回答内容"}}

注意：
- 如果调用工具，必须使用 tool_calls 数组
- 工具执行结果会由后端提供，你不能自己猜测工具结果
"""
  • 协议只有两种合法输出：tool_calls 数组或 final 答案——状态机只有两条出边，越简单模型越守规矩。
  • 「不能编造工具结果」「结果由后端提供」是在堵一个真实漏洞：不写这句，小模型经常自问自答，假装工具已经跑过了。
  • 模型毕竟不是编译器，输出可能带 ```json 围栏、前后闲聊。配套的 _parse_agent_json 做了防御式解析：剥围栏、正则抠 JSON、逐层校验，解析失败就把原文当最终答案（降级而不是崩溃）。

优点：任何会输出 JSON 的模型都能跑，包括本地 4B 小模型。缺点：协议占 prompt 篇幅、解析要防御、模型可能不守格式。于是有了第二种。

SECTION 5.3协议二：native function calling —— 让推理层管格式

OpenAI 协议原生支持工具调用：工具的 JSON Schema 放进请求的 tools 字段，模型想调工具时，响应里不是文本，而是结构化的 message.tool_calls；执行结果用 role: "tool" 的消息回填。格式由推理层约束（constrained decoding），基本不会坏。

prompt 协议native function calling
工具说明书位置system prompt 正文请求体 tools 字段
模型决策形态自由文本里的 JSON（要防御解析）结构化 tool_calls 字段（推理层保格式）
结果回填角色伪装成 user 消息标准 role: "tool" + tool_call_id
模型兼容性任何能输出 JSON 的模型需要模型和推理服务都支持
可靠性中（防御解析兜底）高

本项目把协议做成配置项（agent_tool_protocol: "prompt" | "native"），一套循环两种协议跑，切换是一行配置。这个设计的面试价值：说明你理解「协议」和「编排」是两层可以解耦的东西。

SECTION 5.4读代码：run_agent_events 主循环

核心函数是个 async 生成器（第 1、2 章的 yield 第三次登场）：一边跑循环一边 yield ("event", ...) 直播进度，最后 yield ("result", ...) 收尾。流式和非流式入口共用它——非流式把事件全部抽干只留 result，流式把事件翻译成 SSE 帧。一份状态机，两种消费方式。

for round_index in range(3):                       # ① 硬上限 3 轮
    yield ("event", {"type": "thinking", "round": round_index + 1})

    completion = await create_chat_completion_full(config, request_payload)
    round_usage = completion.get("usage")           # ② 每轮token记账
    if round_usage:
        for key in total_usage:
            total_usage[key] += int(round_usage.get(key) or 0)

    if use_native:
        decision = _decision_from_native(assistant_message)
    else:
        decision = _normalize_agent_decision(_parse_agent_json(raw))

    add_agent_step(run_id=run_id, step_index=step_index,   # ③ 每步落库
                   step_type="model_decision",
                   output_data={"raw": raw, "decision": decision, "usage": round_usage})

    if decision.get("action") == "final":           # 出口①：最终回答
        finish_agent_run(run_id, final_answer)
        yield ("result", _agent_result(...))
        return

    for tool_call in tool_calls:                    # ④ 逐个执行工具
        yield ("event", {"type": "tool_call", "name": tool_name, ...})
        try:
            if tool_name == KNOWLEDGE_TOOL_NAME:
                output = await _run_knowledge_search(...)   # RAG 作为工具复用
            elif tool_name == WEB_SEARCH_TOOL_NAME:
                output = await search_web(...)
            else:
                output = run_tool(tool_name, tool_input)    # 白名单分发
            step["ok"] = True
        except (ToolError, ZeroDivisionError, OverflowError) as exc:
            output = {"error": str(exc)}            # ⑤ 失败也回填，让模型自己调整
            step["ok"] = False
        yield ("event", {"type": "tool_result", "name": tool_name, "ok": ..., ...})

    # ⑥ 回填：两种协议两种姿势（见下）

final_answer = "Agent 工具调用超过最大步数，已停止。..."   # 出口②：兜底
  • ① 硬上限：和第 4 章 Grader 的「最多补检一次」同一哲学——LLM 驱动的循环必须有确定性出口，否则延迟、成本、行为都不可控。
  • ③ 全程落库：每个决策/工具调用/结果都写进 agent_runs / agent_steps 表，包括模型的原始输出 raw。线上「Agent 抽风了」的问题，翻表就能复盘每一步——这是第 7 章可观测性的数据地基。
  • ⑤ 错误回填：工具抛错不终止循环，而是把 {"error": ...} 作为结果喂回去。模型看到「除数为零」经常能自己改参数重试——错误信息本身就是给模型的反馈信号。注意 except 列表是具体异常而不是裸 except Exception：意料中的失败（表达式非法）温柔处理，意料外的 bug 该炸就炸。

第 ⑥ 步的回填是双协议差异的落点，值得单独看：

if use_native and assistant_message is not None:
    # native：标准 function-calling 回填
    messages.append({"role": "assistant", "content": ...,
                     "tool_calls": assistant_message.get("tool_calls") or []})
    for index, tool_result in enumerate(tool_results):
        tool_message = {"role": "tool",
                        "content": json.dumps(tool_result["result"], ensure_ascii=False)}
        if call_id:
            tool_message["tool_call_id"] = call_id   # 结果↔调用 一一对应
        messages.append(tool_message)
else:
    # prompt：assistant 原文 + 伪装成 user 的工具结果
    messages.append({"role": "assistant", "content": raw})
    messages.append({"role": "user", "content":
        "工具执行结果如下：\n" + json.dumps(tool_results, ensure_ascii=False) +
        "\n请基于工具结果继续。仍然只能输出一个 JSON 对象。"})
  • native 路径的 tool_call_id 很关键：一轮里模型可能同时调多个工具，id 让它知道哪个结果对应哪次调用。
  • prompt 路径把结果装进 user 消息——因为不支持 function calling 的模型也不认识 role: "tool"。末尾再敲打一遍「只能输出 JSON」，因为多轮之后模型容易忘记最初的指令。

SECTION 5.5工具安全边界：为什么 calculator 不用 eval()

工具是 Agent 里唯一真正产生副作用的地方，安全边界全在这里。最有代表性的是计算器——最省事的实现是 eval(expression)，而模型的输出本质是不可信的用户输入：一句 __import__('os').system('rm -rf /') 就是远程代码执行。本项目的做法：

_BINARY_OPERATORS = {                       # 白名单：只认这7种运算
    ast.Add: operator.add,   ast.Sub: operator.sub,
    ast.Mult: operator.mul,  ast.Div: operator.truediv,
    ast.FloorDiv: operator.floordiv, ast.Mod: operator.mod, ast.Pow: operator.pow,
}

def _safe_calculate(expression: str) -> int | float:
    if len(expression) > 120:
        raise ToolError("Calculator expression is too long")
    tree = ast.parse(expression, mode="eval")     # 只解析成语法树，不执行
    return _eval_node(tree.body)                  # 手写遍历器

def _eval_node(node: ast.AST) -> int | float:
    if isinstance(node, ast.Constant) and isinstance(node.value, (int, float)):
        return node.value                          # 数字字面量 ✓
    if isinstance(node, ast.BinOp):                # 二元运算 ✓（仅白名单内）
        operator_func = _BINARY_OPERATORS.get(type(node.op))
        if operator_func is None:
            raise ToolError("Calculator operator is not allowed")
        return operator_func(_eval_node(node.left), _eval_node(node.right))
    # 函数调用、属性访问、名字引用……统统不在分支里 → 直接拒绝
    raise ToolError("Calculator expression is not allowed")
  • 思路：ast.parse 把表达式变成语法树（不执行），然后自己写遍历器，只认识数字和白名单运算符。函数调用、属性访问、变量名在遍历器里没有对应分支，落到兜底 raise——默认拒绝（deny by default），比枚举「危险的东西」可靠得多。
  • 长度限制 120 字符防 9**9**9 这类算力炸弹（配合 OverflowError 捕获）。
  • 这段代码是面试讲安全的黄金素材：从「LLM 输出 = 不可信输入」推到「解析而不执行 + 白名单遍历 + 默认拒绝」，一条完整的安全思维链。

其他工具（analyze_log、summarize_git_diff、summarize_ticket、generate_test_cases）共享同一个设计模式：纯函数 + 正则/解析 + 结构化输出 + 给模型的 hint 字段。比如 analyze_log 返回级别统计、错误样本、异常类型清单，最后附一句 "hint": "请基于以上结构化信息，分析最可能的根因..."——工具不光给数据，还提示模型下一步怎么用数据。

SECTION 5.6过程直播：Agent 步骤的 SSE 事件流

Agent 一跑十几秒，用户盯着空屏会以为挂了。本项目把循环里 yield 的事件翻译成自定义 SSE 帧，前端实时渲染「正在思考 → 正在调用 calculator → 工具返回 ✓」的步骤条：

data: {"type": "agent_step", "step": {"type": "thinking", "round": 1}}
data: {"type": "agent_step", "step": {"type": "tool_call", "name": "calculator",
       "parameters": {"expression": "(12+8)*3/2"}}}
data: {"type": "agent_step", "step": {"type": "tool_result", "name": "calculator",
       "ok": true, "summary": "{\"result\": 30}"}}
data: {"choices": [{"delta": {"content": "计算结果是 30..."}}]}   ← 之后是正文流
data: [DONE]
  • 事件帧和正文帧混在同一条 SSE 流里，用 type 字段区分——前端解析器认识 agent_step 就渲染步骤条，不认识就当普通帧忽略，向后兼容。
  • 产品直觉（你的前端优势）：过程可见性不是锦上添花，它把「黑盒等待」变成「透明执行」，用户对同样的 15 秒的忍耐度完全不同。

面试官视角 · 本章问答

Q1用你自己的话说：什么是 Agent？▸

「LLM 在一个有界循环里反复做结构化决策：每轮模型输出『调用某个工具』或『给出最终答案』，工具由后端按白名单执行，结果回填对话进入下一轮。三个工程关键词：有界（我的实现最多 3 轮，超限兜底退出）、白名单（模型只能声明调用，执行权在我手里）、回填(工具结果和错误都进上下文，模型据此自我修正)。它不是自主意识，是一个由 LLM 当决策器的状态机。」

Q2prompt 协议和 function calling 你都实现了，为什么？差异是什么？▸

「差异三点：工具说明书的位置（prompt 正文 vs 请求 tools 字段）、决策的形态（自由文本 JSON 要防御解析 vs 结构化 tool_calls 字段推理层保格式）、回填的角色（伪装 user 消息 vs 标准 role=tool + call_id）。都实现的原因：私有化场景模型不可挑——客户可能跑一个不支持 function calling 的老模型或小模型，prompt 协议是保底兼容层；支持 native 的模型则走 native 拿更高可靠性。协议做成配置项，一套循环两种跑法，这也证明了编排逻辑和协议是可以解耦的两层。」

Q3Agent 的安全风险有哪些？你怎么防？▸

「把模型输出当作不可信用户输入，是一切的出发点。我的分层防御：执行层——工具白名单分发，未注册的名字直接 ToolError；calculator 用 ast.parse 解析成语法树后手写遍历器求值，只认数字和七种运算符，函数调用/属性访问默认拒绝，外加 120 字符长度限制防算力炸弹；循环层——3 轮硬上限防失控；审计层——每一步含模型原始输出全部落库，可复盘。没做但知道要做的：工具级权限分级（读/写分离）、web_search 结果的注入防护（外部网页内容可能包含针对模型的指令）。」

追问预判：「为什么不用 eval + 正则过滤危险词？」→ 答：黑名单永远列不全（编码绕过、字符串拼接）；白名单 + 默认拒绝才是正解。

Q4工具调用失败怎么处理？直接报错给用户吗？▸

「不。失败结果 {"error": "..."} 照样回填给模型，循环继续——模型看到『表达式语法错误』经常自己改写参数重试成功，错误信息就是反馈信号。工程细节：except 只捕获预期异常（ToolError、ZeroDivisionError、OverflowError），意外 bug 不吞，该炸就炸进日志。每次失败也落库（ok=0 + error 字段），评测时能统计各工具的失败率。」

Q5多个工具怎么让模型选对？工具描述怎么写？▸

「工具描述就是给模型看的 API 文档，我的写法有三个套路：说清何时用——每个 description 都带『当用户粘贴日志/报错堆栈时使用』这种触发条件，而不只是功能描述；参数给示例——calculator 的参数描述里直接放 '(12 + 8) * 3 / 2'；输出带 hint——工具返回结构化数据的同时附一句『请基于以上信息分析根因』，引导模型正确消费结果。工具数量也克制在 8 个以内——工具越多选错率越高，说明书越长 prompt 越贵。」

Q6你的 Agent 步骤为什么要全部落库？▸

「三个消费方：调试——『Agent 刚才为什么抽风』翻 agent_steps 表就能看到每轮模型的原始输出和解析后的决策，包括失败的工具调用；评测——离线统计工具选择正确率、平均轮数、每轮 token 消耗（usage 也在 output_json 里）；产品——前端的步骤回放和 run 详情页直接读这两张表。存储上一张 runs 主表 + steps 从表级联删除，一次 run 十几行，成本可忽略。这是 LLMOps 里 tracing 的最小实现，思想和 LangSmith 一致。」

Q7为什么上限是 3 轮？复杂任务 3 轮不够怎么办？▸

「3 是针对我的场景（运维问答 + 轻量工具）压出来的经验值：本地小模型一轮 5–15 秒，3 轮已是用户忍耐上限；观测数据里 90% 的任务 1–2 轮完成，3 轮跑满通常意味着模型在打转而不是在进步。复杂任务的正确解法不是加圈数，是换编排结构：任务分解成子任务分别跑短循环（plan-and-execute 模式），或引入 LangGraph 这类显式状态机做多节点流程——这是我 roadmap 里已经做了原型的方向。」

本章小结 · 装进口袋的 6 句话
  • Agent = 有界循环 + 结构化决策 + 白名单执行 + 结果回填，不是魔法。
  • 模型只 dispatch action，副作用永远在你的代码里发生（redux 直觉）。
  • prompt 协议保兼容（任何模型），native 协议保可靠（推理层管格式），一套循环两种协议。
  • LLM 输出 = 不可信输入：AST 白名单 + 默认拒绝 + 长度限制，不用 eval。
  • 工具错误回填而非终止——错误信息是模型自我修正的反馈信号。
  • 每步落库（含原始输出与 usage）+ SSE 直播步骤 = 可观测的 Agent。
