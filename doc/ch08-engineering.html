SECTION 8.1AI 应用怎么写单测：mock 掉不确定性

难题：核心链路里有个 LLM，输出每次都不一样，还得要 GPU/网络——按传统思路这代码「不可测」。解法是换个问题：你要测的从来不是模型聪不聪明，而是你自己的代码对不对。把系统切成两半：

确定性的一半（真测）概率性的一半（mock 掉 / 交给评测）
切片算法、CJK 分词、RRF 融合排序、JSON 防御解析、工具白名单分发、SSE 帧组装、AST 计算器模型生成的内容质量 → 第 7 章的评测集管；单测里模型被替换成「录音机」

「录音机」怎么做？看 Agent 循环的真实测试——预设一段模型剧本，验证编排逻辑的每一步：

def _fake_completion_factory(responses: list[str]):
    """按顺序回放预设的"模型台词"，形状和真实 completion 一致。"""
    calls = {"count": 0}
    async def fake(config, payload):
        index = min(calls["count"], len(responses) - 1)
        calls["count"] += 1
        return {
            "choices": [{"message": {"role": "assistant", "content": responses[index]}}],
            "usage": {"prompt_tokens": 10, "completion_tokens": 5, "total_tokens": 15},
        }
    return fake

@pytest.mark.anyio
async def test_run_agent_events_yields_progress_then_result(monkeypatch):
    monkeypatch.setattr(agent_runner, "create_chat_completion_full",
        _fake_completion_factory([
            '{"tool_calls":[{"name":"calculator","parameters":{"expression":"6*7"}}]}',
            '{"action":"final","answer":"结果是 42"}',
        ]))

    events = [...]   # 抽干生成器，收集全部事件
    types = [event["type"] for event in events]
    assert types == ["thinking", "tool_call", "tool_result", "thinking"]
    assert result["answer"] == "结果是 42"
    # 两轮模型调用、每轮15 token，验证跨轮usage累加逻辑
    assert result["usage"] == {"prompt_tokens": 20, "completion_tokens": 10, "total_tokens": 30}
  • monkeypatch.setattr ≈ Jest 的 jest.spyOn().mockImplementation()：运行时把 agent_runner 模块里的模型调用函数换成假的，测试结束自动还原。
  • 剧本第一句让模型「调计算器」，第二句「给最终答案」——于是循环的完整路径（决策→工具→回填→再决策→终止）被确定性地走了一遍。工具是真跑的（calculator 真算了 6*7），只有模型是假的。
  • 断言到 usage 累加这种细节——第 3 章那个「流式 usage 丢帧」的坑，就是靠这类测试防回归的。

再看测试的环境隔离（对应 Jest 的 setup 文件）：

"""Test bootstrap: sandbox the SQLite data dir before any server import."""
_TEST_DATA_DIR = tempfile.mkdtemp(prefix="myopenweb-test-")
os.environ["MYOPENWEB_DATA_DIR"] = _TEST_DATA_DIR    # 必须在 import server 之前

import pytest
from server.db import init_db

@pytest.fixture(scope="session", autouse=True)
def _database() -> None:
    init_db()
  • conftest.py 是 pytest 的约定文件（≈ Jest 的 setupFiles）：同目录所有测试自动加载。这里把数据目录指到临时文件夹——测试永远不碰真实数据库。注释里那句「must run first」是真实教训：server.db 在 import 时就解析路径，环境变量晚一步就白设了。
  • fixture ≈ Jest 的 beforeAll，scope="session" 表示整个测试会话只跑一次，autouse 免去每个测试手动声明。
  • 同样的沙箱思路复用到了评测（MYOPENWEB_DATA_DIR=server/eval/.data）和 Docker（/data 卷）——一个环境变量，三处受益，这是第 2 章分层里 db.py 预留的接缝。

SECTION 8.2CI：GitHub Actions 双车道

on: push/PR → main

jobs:
  backend:                      # 车道1：后端
    - setup-python 3.12 (cache: pip)
    - pip install -r server/requirements-dev.txt
    - python -m pytest server/tests -q        # 全部单测，无外部依赖

  frontend:                     # 车道2：前端
    - pnpm install --frozen-lockfile
    - pnpm build                              # tsc 类型检查 + vite 构建
  • 两个 job 并行互不阻塞。后端车道能在 CI 裸容器里跑通的前提正是 8.1 的设计：测试不需要 Ollama、不需要 GPU、不需要网络——可 mock 的边界 = 可 CI 的系统。
  • 前端车道 pnpm build 隐含 tsc——类型错误挡在合并前，等价于一层静态测试。--frozen-lockfile 保证依赖和本地完全一致（你熟）。
  • 面试一句话：「每次 push，12 个测试文件 + 前端类型检查全绿才算数——AI 项目也按正经软件工程来。」

SECTION 8.3Docker：多阶段构建与「模型不进镜像」

# ─── Stage 1: 构建 H5 前端（这一阶段最终会被丢弃）───
FROM node:18-alpine AS web
WORKDIR /app
RUN npm install -g pnpm@10
COPY package.json pnpm-lock.yaml ./
RUN pnpm install --frozen-lockfile     # 先拷清单再装依赖：层缓存关键手法
COPY index.html vite.config.ts tsconfig.json ... ./
COPY src ./src
RUN pnpm build                         # 产出 dist/

# ─── Stage 2: 运行时（最终镜像只有这一层）───
FROM python:3.12-slim
WORKDIR /app
COPY server/requirements.txt ./server/requirements.txt
RUN pip install --no-cache-dir -r server/requirements.txt
COPY server ./server
COPY --from=web /app/dist ./dist       # 只把构建产物搬过来，Node 整套不带

ENV MYOPENWEB_DATA_DIR=/data           # 所有持久状态收敛到一个可挂载目录
VOLUME /data
EXPOSE 8000
CMD ["uvicorn", "server.main:app", "--host", "0.0.0.0", "--port", "8000"]
  • 多阶段：Stage 1 用 Node 镜像编译前端，Stage 2 只 COPY --from=web 拿走 dist——最终镜像里没有 node_modules、没有 Node 运行时，体积从 1GB+ 降到几百 MB。
  • 层缓存：先 COPY 依赖清单、RUN install，再 COPY 源码——改业务代码不触发依赖重装，重构建从几分钟降到几十秒。和你前端 CI 缓存 node_modules 是同一个心智。
  • 单容器 = API + 前端：第 2 章 main.py 那段 StaticFiles 挂载在这里闭环——API 路由优先、其余落 SPA，一个镜像交付整个产品。
  • 状态外置：SQLite + 上传文件全在 /data 卷里，容器可随意销毁重建——十二要素应用的「无状态进程」原则。
为什么模型不进镜像

三个理由：模型权重动辄几 GB，镜像会失控；模型升级频率和代码不同步，耦合发布是灾难；GPU 驱动/推理框架的兼容矩阵复杂，交给专门的 Ollama 容器/服务。所以 compose 里应用容器通过 PROVIDER_BASE_URL 环境变量指向模型服务——模型是外部依赖，和数据库同级。这也正是 providers.py 协议归一在部署层的回报：模型服务随便换，应用镜像不动。

SECTION 8.4收拢一个主题：优雅降级的工程哲学

回看全书，「挂了怎么办」在每一层都有同一个答案——降级继续跑，别把整条链路拖死：

场景降级行为出处
SQLite 没编译 FTS5混合检索退回纯向量db.py _init_fts（第 2 章）
rerank 权重加载失败返回 None，沿用 RRF 粗排rag.py（第 4 章）
WSL 里 localhost 不通探测网关生成候选配置重试providers.py（第 3 章）
Agent 输出的 JSON 解析失败原文当最终答案返回agent_runner.py（第 5 章）
工具执行抛错错误回填给模型自我修正agent_runner.py（第 5 章）
检索结果为空切换拒答 prompt，不裸答rag.py（第 4 章）
LLM 标题/追问生成失败静默跳过，不影响主对话llm_tasks.py

面试收拢话术：「AI 应用的依赖天生不稳定——模型会超时、权重会缺失、输出会不合法。我的原则是每个增强能力都是可降级的插件，核心链路（问答）在任何单点故障下保持可用。」一张表七个例子，比任何形容词都有力。

面试官视角 · 本章问答

Q1LLM 应用的输出不确定，你怎么写测试？▸

「把系统切成确定性和概率性两半。确定性的（切片、RRF 排序、JSON 解析、工具分发、SSE 组帧）占代码量大头，正常单测；模型调用在测试里用 monkeypatch 换成剧本回放的 stub——形状和真实响应一致，内容预设。这样 Agent 循环的『决策→工具→回填→终止』全路径可以确定性验证，包括 usage 累加、错误回填这些细节。模型『答得好不好』不归单测管，归第 7 章的评测集管。12 个测试文件在 CI 裸容器里跑，零外部依赖。」

Q2讲讲你的 Docker 部署方案，为什么这么设计？▸

「多阶段构建：Node 阶段编译 H5，运行时阶段只拿 dist，最终镜像是 python-slim + 后端 + 静态产物，没有 Node。API 和前端一个容器交付——FastAPI 挂 StaticFiles，API 路由优先、其余落 SPA，内网私有化场景一个 docker compose up 起完。状态全部收敛到 /data 卷（SQLite + 上传文件），容器无状态可随意重建。模型不进镜像，通过环境变量指向外部 Ollama/vLLM 服务——权重体积、升级节奏、GPU 兼容都和应用解耦。」

追问预判：「为什么不前后端分两个容器 + nginx？」→ 答：单用户内网工具，一个容器是运维成本最低的交付形态；规模化时 dist 丢 CDN/nginx、后端横向扩，架构上已经是分开的（前端纯静态），拆分是零改动的。

Q3什么是优雅降级？举你项目里的例子。▸

「增强能力失效时系统退到基础能力继续服务，而不是报错终止。我的项目里成体系：rerank 权重缺失→沿用 RRF 粗排；FTS5 不可用→纯向量检索；Agent JSON 解析失败→原文当答案；工具抛错→错误回填让模型修正；检索为空→切拒答 prompt。设计模式是统一的：增强层返回 None/错误信号，调用方判空走基线路径——每个 AI 能力都是可拔插件，核心问答链路在任何单点故障下可用。AI 应用的依赖（模型、权重、网络）天生比传统服务不稳定，降级不是可选项而是默认设计。」

Q4你的 CI 里为什么不跑评测集？▸

「评测要真模型（Ollama + bge-m3 + 生成模型），CI 裸容器没有 GPU 也不该拉几个 GB 的权重——所以分层：CI 管确定性（单测 + 类型检查，秒级、每次 push）；评测管质量（本地/专用机器手动或定时触发，分钟级、改检索相关代码时跑）。这也是业界主流分法：质量回归做成 nightly job 或发布前门禁，而不是塞进每次 commit 的 CI。如果公司有 GPU runner，我会把小评测集挂到 PR 门禁上，大集 nightly。」

Q5SQLite 用在生产靠谱吗？什么时候必须换？▸

「看负载形态。我的场景——内网单用户/小团队工作台，读多写少、并发个位数——SQLite 是最优解：零运维、零网络开销、备份就是拷文件，还白送 FTS5 全文检索。必须换的信号：多实例部署（SQLite 是进程内库，无法跨机共享写）、写并发上百、需要行级权限。演进路径不是嘴上说说：向量这块已经通过 VectorStore 抽象落地了 pgvector 可切换后端（双后端契约测试在 CI 跑真 PG），业务表要换 PostgreSQL 时同样只动 repositories/。技术选型答『匹配当前约束 + 已验证的演进路径』永远比报菜名强。」

本章小结 · 装进口袋的 5 句话
  • AI 应用测试 = 确定性逻辑真测 + LLM 边界 mock 成剧本回放；质量归评测，正确归单测。
  • conftest.py 沙箱数据目录 ≈ Jest setupFiles；monkeypatch ≈ mockImplementation。
  • CI 双车道并行：pytest 全绿 + tsc/vite 构建通过才能合并；可 mock 的边界 = 可 CI 的系统。
  • Docker 多阶段：Node 只在构建期，镜像无 node_modules；状态收敛 /data 卷；模型是外部依赖不进镜像。
  • 优雅降级是体系不是技巧：七处降级一个模式——增强层可拔插，核心链路永远可用。
