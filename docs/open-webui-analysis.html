Open WebUI 对照分析
—— 拆解参考项目的架构原则与借鉴边界

分析对象：Open WebUI v0.8.12。抽取它的架构原则、划定「个人版最小闭环」的借鉴边界——本项目多数核心设计决策的源头。

◈
阅读提示 · 历史文档

这是项目早期的拆解笔记，文中「建议下一步 / 待完成」的事项（tool_calls 格式统一、Agent 运行日志、Memory、文件落盘、RAG、原生 function calling、MCP Server、PostgreSQL 向量后端等）此后已全部落地。保留原文是为了追溯设计决策的来龙去脉——每个功能"为什么这么做"的第一现场。

结论

Open WebUI 是完整平台，不是单纯聊天 App。它的核心复杂度集中在：

  • 多 provider / 多模型路由
  • 用户、权限、分享、分组
  • 工具注册、工具权限、工具调用循环
  • 文件、知识库、RAG、引用来源
  • 记忆系统
  • WebSocket 事件流
  • Pipeline / Function 插件系统
  • 多数据库迁移和大量配置项
◈
路线判断

MyOpenWeb 定位是个人版 AI App 骨架，不能整包照搬。合理路线是：抽取 Open WebUI 的架构原则，但只实现个人版最小闭环。

模型与聊天路由

相关文件：utils/chat.py、routers/openai.py、routers/ollama.py、functions.py、utils/middleware.py。

Open WebUI 的聊天请求不是直接转发——它会先经过统一聊天入口，再按模型归属分发到 Ollama / OpenAI / Function / Pipeline 等不同路径。核心思想：

请求
→ 统一 chat completion 入口
→ 读取模型配置
→ 权限检查
→ pipeline/filter 处理
→ 工具 / RAG / memory 注入
→ 分发到真实 provider
→ 处理流式输出和事件

MyOpenWeb 实现了其中的最小版：

/api/chat/completions
→ ProviderConfig
→ Ollama / OpenAI Compatible
→ 统一 OpenAI SSE

当时的判断：不需要照搬权限和 pipeline，应该先补「可观测性」和「工具调用日志」——这两项随后成为 agent_runs/agent_steps 的设计来源。

Tools / Agent 机制

相关文件：utils/tools.py、models/tools.py、routers/tools.py、tools/builtin.py、utils/middleware.py、config.py。Open WebUI 有两类工具调用方式：

方式 A：Prompt 模拟函数调用

Available Tools: {{TOOLS}}

Return only JSON:
{
  "tool_calls": [
    {"name": "toolName", "parameters": {"key": "value"}}
  ]
}
用户问题
→ 用 task model 判断需要哪些工具
→ 解析 tool_calls
→ 执行工具
→ 把工具结果作为 sources/context 注入后续回答

这和 MyOpenWeb 早期 Agent v1 接近，但格式不同。Open WebUI 用 tool_calls 数组：

{
  "tool_calls": [
    {
      "name": "calculator",
      "parameters": {
        "expression": "(12 + 8) * 3 / 2"
      }
    }
  ]
}

而 MyOpenWeb 当时的格式是单工具对象：

{
  "action": "tool",
  "tool": "calculator",
  "input": {
    "expression": "(12 + 8) * 3 / 2"
  }
}
✓
当时的建议（已落地）

把 MyOpenWeb 改成更接近 Open WebUI 的 tool_calls 数组格式——后续天然支持一次调用多个工具，也更接近 OpenAI function calling 结构。此建议随后实施，且旧格式保留了兼容解析。

方式 B：Native Function Calling

Open WebUI 也支持模型原生 tool calls。在流式响应里它会收集 tool_calls / function.name / function.arguments / tool_call_id，然后执行工具，再把结果作为 tool / function_call_output 消息继续交给模型。

当时的判断是暂缓：Ollama 本地小模型对原生 tool calling 支持不稳定，先把 prompt-tool-loop 做稳更重要，后续再加 native 作为高级路径——后来以 agent_tool_protocol: prompt | native 双协议可切换的形态落地。

Tools 数据模型

tool
├── id
├── user_id
├── name
├── content        # 存工具代码
├── specs          # 存工具 schema
├── meta
├── valves         # 存工具配置
├── created_at
└── updated_at

重点：content 存工具代码、specs 存 schema、valves 存配置，还有权限和 access grants。

◈
借鉴边界

MyOpenWeb 不需要动态上传工具代码——工具实现放在后端白名单 Python 代码里，不允许用户上传任意代码。这是安全模型的分水岭：Open WebUI 是平台（用户可扩展工具），MyOpenWeb 是产品（工具由代码库定义）。

Agent 运行日志

Open WebUI 在流式过程中会向前端发送工具执行事件，并把工具结果合并到输出结构里。这给了用户一个关键能力：看见模型是否调用工具，以及工具执行结果是什么。

MyOpenWeb 当时缺这个能力，这份笔记把它定为最高优先级，并给出了表设计（随后原样落地）：

agent_runs
├── id
├── conversation_id
├── message_id
├── model
├── user_input
├── final_answer
├── created_at
└── updated_at

agent_steps
├── id
├── run_id
├── step_index
├── type              # model_decision / tool_call / tool_result / final
├── name
├── input_json
├── output_json
├── ok
├── error
└── created_at

前端先不做复杂调试面板，先在 assistant 消息下方显示「Agent: 调用了 calculator，结果 30」，后续再做展开详情——最终演进为步骤时间线 + 运行日志回放面板。

Memory 机制

相关文件：models/memories.py、routers/memories.py。Open WebUI 的 memory 分两层：数据库存原始 memory 文本，向量库保存 embedding 用于相似度查询。结构很简单：

memory
├── id
├── user_id
├── content
├── created_at
└── updated_at

MyOpenWeb 的简化版设计（已实现）：

memories
├── id
├── content
├── category       # preference / profile / project / fact
├── enabled
├── created_at
└── updated_at

第一阶段不需要向量库——把所有启用 memory 注入 system prompt 即可，数量多了再做 embedding/RAG。

Files / Knowledge / RAG

相关文件：models/files.py、models/knowledge.py、routers/retrieval.py、retrieval/。Open WebUI 把文件和知识库拆开：

file
├── id
├── user_id
├── hash
├── filename
├── path
├── data
├── meta
├── created_at
└── updated_at

knowledge
├── id
├── user_id
├── name
├── description
├── meta
├── created_at
└── updated_at

knowledge_file
├── knowledge_id
└── file_id

这个拆法值得借鉴（随后被采纳为 MyOpenWeb 的 files / knowledge / knowledge_file 三表结构）。当时 MyOpenWeb 的文件只是临时放在前端消息里，历史附件不可复用。笔记建议的路径：先「文件落盘 + 文本注入 + 可复用历史附件」，再上向量检索——项目实际演进完全遵循了这个顺序。

不建议现在照搬的部分

当时圈定暂时不做的平台能力：

  • 用户系统、复杂权限、Access Grants
  • 动态上传 Python Tool
  • MCP Tool Server、OpenAPI Tool Server
  • Pipeline / Filter 插件
  • 多租户、WebSocket 实时协作
  • 大规模向量库、PostgreSQL 迁移
◈
后记

其中两项后来随项目成熟主动补齐：MCP Server（把知识库暴露为标准 MCP 服务，且做的是 Server 侧而非 Open WebUI 的 client 侧）与 PostgreSQL + pgvector（经 VectorStore 抽象做成可切换后端）。其余仍按「个人版边界」保持不做——边界本身也是设计。
