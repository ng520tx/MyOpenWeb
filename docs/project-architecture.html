DOC-01 · PROJECT ARCHITECTURE

项目架构
—— 从前端直连到分层可演进的全栈骨架

技术栈、目录结构、核心数据流、模块拆解、数据库设计与 Android 壳工程——一份读完即可上手改代码的工程地图。

范围 全栈 后端 FastAPI + SQLite / pgvector 前端 React 18 + TS 5 移动端 Kotlin WebView

项目定位

基于 Open WebUI 拆解的个人版 AI App 骨架，目标是做成「企业研发/运维 AI Copilot 工作台」的求职作品。当前架构已经从「前端直连模型服务」升级为：

  • React H5 聊天页
  • Android WebView 壳
  • FastAPI 本地后端
  • SQLite 对话/配置/Agent 日志/记忆/文件/知识库持久化
  • 原生 STT/TTS/文件桥接
  • Ollama / OpenAI Compatible 模型接入
  • 文件落盘 + 文本抽取（txt/md/pdf/docx，可选 PaddleOCR）
  • 自研 RAG 知识库（切片 + 向量化 + 混合检索 BM25/向量 RRF 融合 + 可选 bge-reranker 重排 + 引用来源 + 无答案拒答）
  • 检索质量评测（40 条 QA，Hit@K / MRR / 延迟参数对照，server/eval/）
  • 研发/运维 Agent 工具（日志分析、Git diff 摘要、工单总结、测试用例生成、知识库检索）
  • pytest 单元测试 + GitHub Actions CI + Docker 一键部署
◈
定位说明

当前阶段的目标不是做成 Dify 那样的工作流平台，而是把 RAG 知识库和真实研发/运维 Agent 工具全部自研落地，便于面试讲清底层。RAG / Agent 功能模块的产品化说明见 D-02 · RAG + Agent 模块。

核心数据流

聊天请求

用户输入
→ MessageInput.onSend(text)
→ ChatPage.handleSend()
→ store.addMessage('user', text, files)
→ store.addMessage('assistant', '')
→ syncProviderConfig(settings)
→ POST /api/chat/completions
→ FastAPI 根据 provider 配置选择真实模型服务
   ├─ Ollama：统一转发原生 /api/chat，并转换成 OpenAI SSE / 标准 completion JSON
   └─ OpenAI Compatible：转发 /chat/completions
→ 前端统一用 createOpenAITextStream() 解析
→ appendContent(aiId, delta)
→ feedStreamTTS(delta)
→ 完成后 persistNow()
→ saveChat(activeConversation)

Provider 配置

SettingsDrawer 修改 providerType / apiBaseUrl / apiKey
→ 保存到 localStorage
→ 用户点击「保存连接配置到后端」或模型刷新/聊天发送时触发 syncProviderConfig()
→ PUT /api/config/provider
→ SQLite.app_config 持久化

对话同步

App 启动
→ fetchProviderConfig()
→ fetchChats()
→ 本地 localStorage 与 SQLite 做按 updatedAt 合并
→ 以较新的版本为准回写到前端 Store
→ 本地较新的记录再回传 SQLite

模块说明

前端 API 层（src/apis/）

  • chat.ts：不再直连第三方模型服务，只请求本地后端 /api/chat/completions；保留文件和图片消息的 OpenAI content-part 组装逻辑
  • config.ts：fetchProviderConfig() 读取后端保存的 provider 配置；syncProviderConfig() 把当前设置同步给后端
  • chats.ts：fetchChats() / saveChat() / removeChat()，把本地对话缓存和 SQLite 连接起来
  • models.ts：统一请求 /api/models，由后端代为访问真实 provider
  • streaming.ts：createOpenAITextStream() 是主路径，createOllamaTextStream() 保留为兼容工具

本地后端（server/）

  • main.py：初始化 SQLite、注册 CORS（MYOPENWEB_CORS_ORIGINS 可收紧）、挂载全部路由
  • db.py：建表 + 外键/WAL/busy_timeout pragma；默认 provider 配置为 Ollama
  • routers/config.py：GET/PUT /api/config/provider（API key 以 ***xxxx 指纹脱敏返回，掩码回传不覆盖原值）；POST /api/config/provider/verify 检测当前 provider 是否可连接，返回配置地址、实际访问地址、模型数量、模型列表、错误信息
  • routers/models.py：GET /api/models
  • routers/chat_proxy.py：POST /api/chat/completions，统一返回 OpenAI SSE，前端只维护一种流解析逻辑；检索失败降级 + retrieval_warning
  • routers/agent.py：POST /api/agent/completions（Agent 入口，复用同一套聊天请求结构，只开放后端白名单工具）；GET /api/agent/runs/{run_id} 查询一次 Agent 运行详情
  • routers/memories.py：GET/POST /api/memories、PUT/DELETE /api/memories/{id}，管理手动长期记忆，Agent 请求时自动注入启用记忆
  • routers/chats.py：GET /api/chats、PUT/DELETE /api/chats/{id}
  • services/providers.py：OpenAI Compatible 转发；Ollama 原生 /api/chat 转发；Ollama NDJSON → OpenAI SSE 协议转换；WSL 下自动尝试 localhost 和 Windows 主机网关地址，避免 Ollama 运行在 Windows 时本地后端连不上
  • services/agent_tools.py：get_current_time（服务器时间）、calculator（AST 安全求值，不用 eval）、analyze_log（日志级别统计/错误行/异常类型）、summarize_git_diff（变更文件与增删行数）、summarize_ticket（工单号/@相关人/链接/字段抽取）、generate_test_cases（测试覆盖维度）、search_knowledge（仅选定知识库时开放，由 agent_runner 异步调用检索）
  • services/file_extract.py：txt/md 等文本直读，pdf 用 pypdf，docx 用 python-docx，缺依赖时报清晰错误而不崩溃
  • services/embeddings.py：批量向量化，复用 providers.py 的 URL 解析与 WSL 回退
  • services/rag.py：切片（按字符 + 自然边界 + overlap）、批量向量化后写入 VectorStore；检索时向量 top-k + 关键词 top-k 由所选后端提供，本层做 RRF 融合 / rerank / 自纠错；维度不匹配（换了 embedding 模型）时返回空并提示重建索引；聊天融合拼接带来源标注的参考资料 system_prompt，未命中时走拒答提示
  • vectorstores/（向量存储抽象层）：base.py 定义 VectorStore 协议（replace/count/delete + query_by_vector / query_by_keywords 两个排序原语）；sqlite_store.py 默认实现（JSON 向量 + numpy 内存余弦 + 进程内矩阵缓存与版本戳失效 + FTS5 BM25）；pgvector_store.py（ORDER BY embedding <=> q 余弦 + tsvector 关键词检索）；factory.py 按 MYOPENWEB_VECTOR_BACKEND + MYOPENWEB_PG_DSN 选择实现
  • services/agent_runner.py：要求模型输出 Open WebUI 风格的 tool_calls 数组；兼容小模型把工具名直接写进 action 字段的情况；根据模型决策调用白名单工具并把结果回填给模型；工具循环轮数可配置（agent_max_rounds，默认 3，clamp 1-10）；知识库检索失败时工具返回结构化错误（模型转告用户），运行不中断；写入 agent_runs / agent_steps；注入已启用 memories 作为长期上下文

状态管理（src/stores/）

使用单一 useAppStore 管理：conversations、activeConversationId、generating、settings、pendingFiles、sidebarOpen、settingsOpen。

持久化策略：

  • 设置先写 localStorage，Provider 配置再同步到 SQLite
  • 对话先写 localStorage，流结束后同步到 SQLite
  • 文件内容在本地持久化时仍然剥离，避免 localStorage 膨胀

设置面板（src/components/settings/SettingsDrawer.tsx）

  • Provider 连接配置：providerType / apiBaseUrl / apiKey，可手动保存到后端，可点击「测试连接」检查 provider 是否可访问
  • Agent 配置：agentEnabled（开启后聊天请求走 /api/agent/completions）、工具调用协议（prompt / native）、最大工具轮数（1-10）、联网搜索开关
  • 聊天运行参数：model / systemPrompt / temperature / maxTokens / streamOutput / ttsEnabled / ttsLang / ttsRate

对话与侧边栏（src/components/sidebar/Sidebar.tsx）

  • 支持新建、切换、删除；按标题和内容搜索
  • Markdown / JSON 导出
  • 删除时会同时删除 SQLite 中的记录

文件与图片消息

  • 浏览器环境：<input type="file">；Android WebView：优先走原生 pickFile()
  • 文本文件：作为上下文拼接进消息文本
  • 图片文件：作为 OpenAI content-part 的 image_url；对于 Ollama，后端统一走原生 /api/chat

TTS / STT

  • TTS：使用浏览器 speechSynthesis（流式分句朗读）
  • STT：浏览器优先尝试 SpeechRecognition，Android WebView 回退到原生 NativeSTT

Android Bridge

桥接接口：startSTT() / stopSTT() / playTTS() / stopTTS() / pickFile() / goBack() / setTitle() / getSafeArea()。

  • H5 端统一通过 JSON.stringify(params) 调 callNative()
  • Android 权限回调已回接到 NativeSTT.onPermissionResult()
  • WebView 容器暴露 getBridge() 供 Activity 权限回调使用

Android 壳工程

位于 android/ 子目录，Kotlin + Gradle 独立构建。

技术栈

  • 语言：Kotlin；最低 API 26（Android 8.0），Target API 34（Android 14）
  • 构建：Gradle 8.5 + AGP 8.2.2
  • 纯原生 API，无第三方业务依赖

  • DEBUG：加载开发服务器
  • RELEASE：加载 file:///android_asset/web/index.html
  • 通过 BuildConfig.DEBUG 自动切换

→ NativeSTT / NativeTTS / NativeFilePicker / SafeAreaHelper
→ evaluateJavascript("window[cbFuncName](data)")

演进复盘：五步入门路线（全部完成）

项目按「让本地开发链路可观测 → 用最小闭环理解 Agent 核心结构」的路线演进，五步全部落地，保留于此作为学习路径参考。

第 1 步：稳定启动与连接自检

目标：用户能明确知道是哪一层出问题，而不是只看到前端 HTTP 500。

  • 后端新增 POST /api/config/provider/verify，返回 provider 类型、配置地址、实际访问地址、模型数量、错误信息
  • 设置页新增「测试连接」按钮：成功显示可用模型列表摘要，失败显示明确原因（FastAPI 未启动 / Ollama 未启动 / 模型服务不可达 / API Key 错误）
  • 学习点：FastAPI 路由组织、Pydantic 返回结构设计、前端 loading / success / error 状态处理

第 2 步：Agent v1 最小工具调用循环

目标：从「普通聊天 App」进入「Agent App」。首批安全工具：get_current_time、calculator。

用户输入
→ 后端构造 system prompt，告诉模型有哪些工具
→ 模型判断是否需要调用工具
→ 后端执行工具
→ 把工具结果再交给模型
→ 模型生成最终回答
→ 前端展示
  • Agent 不是神秘能力，本质是「模型 + 工具 + 控制循环」
  • 工具必须由后端白名单控制，不能让模型直接执行任意命令
  • 已兼容 {"action":"calculator","expression":"..."} 这类小模型常见错误格式；主协议为 {"tool_calls":[{"name":"calculator","parameters":{...}}]}

第 3 步：Agent 运行日志

  • 记录用户输入、模型判断、工具名、工具参数、工具返回结果、最终回答
  • 前端 assistant 消息下显示「Agent 调用 N 个工具」，详细步骤经 GET /api/agent/runs/{run_id} 查询
  • 学习点：可观测性比「功能多」更重要；出错时日志能快速定位是模型判断错、工具参数错，还是工具执行错

第 4 步：记忆与偏好

  • 设置页可新增 memory，支持 fact / preference / profile / project 分类与启用/停用/删除
  • Agent 请求自动注入已启用 memory；暂不做 embedding，相似度检索留到 RAG 阶段
  • 学习点：记忆不是把所有聊天塞进 prompt——稳定偏好和长期事实单独存储，每次请求只注入必要记忆

第 5 步：文件知识库 / RAG

  • 文件落盘 → 文本切分 → 向量索引 → 相似度检索 → 检索结果注入 prompt，全链路已实现并持续演进（混合检索 / rerank / 自纠错 / pgvector）
  • 学习点：RAG 是「检索 + 上下文注入」，不是简单上传文件；文件、分块、索引、引用来源要分开设计
