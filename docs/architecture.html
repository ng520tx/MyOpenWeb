RAG 检索链路

Retrieval Pipeline · 每一步可开关、可降级、可评测
STEP 01
文档切片向量化
txt/md/pdf/docx 抽取（可选 OCR），chunk_size=600 重叠切片，bge-m3 向量化落 SQLite
STEP 02 · 可选
多轮改写
"它的端口是多少" → 结合历史改写为自包含查询，失败回退原文
STEP 03
混合检索
BM25（FTS5 + 自研 CJK 二元组分词）与向量余弦并行召回
STEP 04
RRF 融合
两路排名按倒数排名融合，兼顾术语精确匹配与语义泛化
STEP 05 · 可选
Rerank 重排
bge-reranker cross-encoder 精排，未装依赖自动降级
STEP 06 · 可选
Grader 自纠错
评估资料充分性，不足则按缺口重检索一轮合并（上限 1 次）
STEP 07
引用生成 / 拒答
强约束 prompt 只依据资料回答，[序号] 标注来源；未命中明确拒答
降级哲学
rerank / 改写 / Grader 任一环节失败或超时，自动回退上一环节结果，检索主链路永不阻塞。
中文 BM25
FTS5 unicode61 无法切中文，自研 CJK 字符二元组 + 英文小写分词，零额外依赖让 BM25 在中文语料生效。
防幻觉
库外拒答 + 引用展示 + embedding 维度变更拒用旧索引，三层防线控制编造。

06

关键技术决策

Design Decisions · 每个取舍都有理由和退路
D1

为什么自研 RAG / Agent，而不用 LangChain / Dify？

目标是把切片策略、相似度计算、拒答规则、工具循环上限、运行日志的每一步做成可解释、可调试的白盒。repositories 层已预留切换点，规模上来按接口替换即可；另有 examples/langgraph-agent 用 LangGraph 复刻同等能力沉淀对比结论——不是不会用框架，是先懂原理再选框架。

D2

向量存储：SQLite 默认，pgvector 可切

检索只依赖 VectorStore 接口（向量 top-k + 关键词 top-k），默认 SQLite 存 JSON 向量 + numpy 内存余弦（百级文档毫秒级、零额外服务）；MYOPENWEB_VECTOR_BACKEND=pgvector 一个环境变量切到 PostgreSQL（ORDER BY embedding <=> :q + tsvector 关键词），同一套契约测试保证双后端行为一致——"规模上来按接口替换"是已验证的路径，不是口头预案。

D3

全链路优雅降级

rerank 依赖（torch）独立隔离，未安装自动回退融合排序并做负缓存；检索链路挂掉（embedding/向量后端不可达）时聊天降级为普通回答并给前端黄色警示而不是 502；query rewrite / Grader / 联网搜索 / 标题生成任一失败均回退或静默——增强环节永远不能拖垮核心链路，这是生产可用与 demo 的分水岭。

D4

工具安全边界

工具由后端白名单实现，模型只能表达"调用意图"，不能执行任意代码；计算器用表达式解析器而非 eval；知识库约束下强制先检索、未命中拒答。代码解释器类高危能力刻意不做，讲清"为什么不做"。

D5

协议归一：前端只有一条流解析路径

Ollama NDJSON、OpenAI SSE 在后端 providers 统一转换为 OpenAI SSE（含 usage 帧、agent_event 帧扩展字段），前端 streaming.ts 单一解析器；换模型服务商前端零改动。

D6

评测先行，数据说话

每个检索增强特性都有开/关对照评测再合入：混合检索 +11pp、改写 +26pp、自纠错在困难集 +9pp 而标准集持平（如实呈现适用边界）。评测脚本与语料全部入库，server/eval/ 一条命令复现。
